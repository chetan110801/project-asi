# SELF-IMPROVEMENT SYSTEM
**How the system upgrades itself.** This is the part that makes the project *living* instead of static. The roadmap improves the territory; this improves the *roadmap-maker*.

`Part of: PROJECT ASI — Living Instruction System`
`System Version: 1.0 · Status: Living document · Last updated: 2026-06-20`

---

## 1. The idea

> A normal project gets better at its task. A *self-improving* project also gets better at **getting better**.

So every cycle improves two layers:
- **Layer 1 — the knowledge** (what we understand about intelligence).
- **Layer 2 — the system** (how we research, learn, and decide).

Neglecting Layer 2 is how projects quietly rot: working hard with dull tools.

---

## 2. The improvement loop

```
1. OBSERVE   What worked, what didn't, what was slow, what was confusing?
2. DIAGNOSE  Why? Was it the method, the document, a missing domain, a bad habit?
3. PROPOSE   A specific change to a specific document or process.
4. WEIGH     Benefits, costs, trade-offs, and what it might break.
5. APPLY     Make the change. Bump the version. Record it.
6. WATCH     Did it actually help next cycle? If not, revert or revise.
```

Step 6 matters: a "improvement" that doesn't help is just a change. We keep only changes that earn their place.

---

## 3. What triggers an improvement

- A **repeated friction** (the same thing keeps being slow or confusing).
- A **surprise** (reality didn't match our expectation — a prediction missed).
- A **contradiction** found between two documents.
- A **blind spot** revealed (a domain or failure mode we'd ignored).
- An **audit** (a deliberate, scheduled self-critique — see [`QUALITY_CONTROL_SYSTEM.md`](QUALITY_CONTROL_SYSTEM.md)).
- A **better idea** for how to work, from anywhere.

---

## 4. Versioning rules

Nothing is final. When we change how the system works:
- give it a **version** (system version in [`MASTER_INSTRUCTION.md`](MASTER_INSTRUCTION.md));
- record it in [`VERSION_HISTORY.md`](VERSION_HISTORY.md) with: what changed, *why*, expected benefit, trade-offs;
- **preserve the history** — we never silently overwrite the past; we can always see how thinking evolved.

Small fixes are minor versions (1.1, 1.2). Big rethinks are major versions (2.0).

---

## 5. Guardrails against bad "improvements"

Change for its own sake is a trap. A proposed change must pass:
- **Does it serve the prime directive** (understanding reality)? 
- **Does it reduce friction or blindness**, or just add complexity?
- **Is it reversible** if it turns out wrong?
- **Does it keep the system simple enough to actually use?** (A system too elaborate to follow is worse than a simple one followed.)

> **Beware "system astronomy":** endlessly polishing the machinery instead of using it. The system exists to produce understanding, not to admire itself. (This risk is so real it has dedicated defenses — see the Milestone & Showcase system and [`QUALITY_CONTROL_SYSTEM.md`](QUALITY_CONTROL_SYSTEM.md).)

---

## 6. The meta-question, asked every cycle

> *"If a much smarter version of this system looked at how we're working now, what's the first thing it would change?"*

Then we consider actually changing it. That single question is the engine of Layer 2.
