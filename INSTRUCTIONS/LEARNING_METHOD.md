# LEARNING METHOD
**How the learner actually learns and remembers.** Research is useless if it doesn't stick in *your* head. This document is about building real, durable understanding — not the feeling of understanding.

`Part of: PROJECT ASI — Living Instruction System`
`System Version: 1.0 · Status: Living document · Last updated: 2026-06-20`

---

## 1. The core distinction

There are two very different things:
- **Familiarity** — "I've seen this before, it feels clear." (Cheap. Fades fast. Often fake.)
- **Understanding** — "I can rebuild this idea myself and use it on a new problem." (Expensive. Lasts. Real.)

Reading and watching mostly produce *familiarity*. This system is built to push past that into *understanding*.

> **The test of understanding:** can you explain it simply, from memory, to someone else — and use it somewhere new? If not, you have familiarity, not understanding.

---

## 2. The four explanation levels (every big idea gets all four)

1. **5-minute version** — the gist, one analogy, why it matters.
2. **30-minute version** — the main parts, how they fit, the key example.
3. **Deep version** — the details, the math/mechanism, the edge cases.
4. **Expert version** — the open debates, the failure cases, what the experts argue about.

You climb the ladder only as far as a topic deserves. Not everything needs the expert level — leverage decides (see [`RESOURCE_COLLECTION_SYSTEM.md`](RESOURCE_COLLECTION_SYSTEM.md)).

## 3. Techniques that actually work (and why)

| Technique | What you do | Why it works |
|---|---|---|
| **Active recall** | Close the book; explain from memory. | Retrieving builds memory far more than re-reading. |
| **Teach-back (Feynman)** | Explain it to an imaginary 12-year-old. | Gaps in understanding become obvious instantly. |
| **Spaced repetition** | Revisit at growing intervals (1 day, 3 days, 1 week…). | Memory strengthens when refreshed just before forgetting. |
| **Interleaving** | Mix related topics instead of one big block. | Builds flexible understanding, not narrow memorization. |
| **Concrete-first** | Start with a real example, then generalize. | Brains grab examples before abstractions. |
| **Build/draw it** | Make a diagram, toy model, or tiny demo. | Forces the fuzzy parts to become exact. |

---

## 4. Tools for understanding hard ideas

- **Analogies** — map a new idea onto a familiar one (then note where the analogy *breaks*, because every analogy lies a little).
- **Mental models** — small reusable "thinking machines" (e.g., feedback loops, bottlenecks, trade-offs).
- **Stories** — wrap an idea in a sequence of events; brains remember stories.
- **Visual thinking** — boxes, arrows, before/after pictures.
- **Comparisons** — "how is this like and unlike the thing I already know?"

---

## 5. The language rule (especially for an ESL learner)

- Hard word appears → define it immediately, in-line, in simple terms, with an example.
- Prefer a short common word over a long fancy one.
- One idea per sentence when the idea is hard.
- Hard words worth keeping go into [`GLOSSARY.md`](GLOSSARY.md) so they're learned once and reused.

This is the "simplify the language, never the idea" rule from the constitution, made practical.

---

## 6. Proof of learning

Understanding is claimed only when you can produce **one** of these from memory:
- a clean 5-minute explanation in your own words,
- a diagram you drew yourself,
- a correct answer to a *new* question using the idea,
- a tiny build/demo,
- a checkable prediction.

These proofs aren't just tests — they're the **showable artifacts** that mark progress. (How progress is tracked and celebrated lives in the Milestone & Showcase system — see [`VERSION_HISTORY.md`](VERSION_HISTORY.md) for when it was added.)
